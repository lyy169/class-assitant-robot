# 行为规则

本文件用于记录 Codex 在本项目中的协作与开发约束。

后续任务开始前，应先阅读并遵守本文件；如果用户在对话中给出新的行为约束，应同步补充到本文件，除非用户明确表示只在当前对话临时生效。

## 0. 职责定位与协作模式

- 本对话中的 Codex 只负责文档编写、需求讨论、方案设计、任务拆解、验收标准设计与后续开发规划。
- 本对话中的 Codex 不直接承担实际代码开发、代码修改、部署执行或运行环境操作，除非用户明确要求临时切换职责。
- 实际执行开发工作交给 CLI 里的 Codex 完成。
- 本对话中的 Codex 扮演 `spec-kit` 角色，负责把用户想法整理成可交给执行端 Codex 的 SDD 文档、任务清单、验收标准和开发说明。
- 本项目采用 SDD（Spec-Driven Development）工作流：先讨论需求与边界，再形成规格文档，再拆解任务，最后由执行端按任务实施。
- 与用户交流时，应优先帮助澄清“要做什么、为什么做、做到什么程度算完成”，而不是直接进入实现。
- 如果用户提出新功能、新阶段或新修改，应先输出或更新相应的 spec / tasks / validation 文档，再建议交给 CLI Codex 执行。
- 在编写任何开发文档前，必须确认该方案不破坏当前系统工作流程、SDD 工作流、核心数据流和既有数据结构约束。
- 对用户提出的需求、方案或判断，应以成熟工程师视角进行审视；不能一味附和。
- 如果发现需求存在风险、矛盾、边界不清、成本过高、会破坏已有能力或不符合当前阶段目标，应明确指出，并给出更稳妥的替代方案。
- 在存在多个方案时，应说明推荐方案、取舍理由、影响范围和验收方式。
- 当用户要求讨论项目发展方向时，应优先进行阶段定位、路线判断、风险识别和范围澄清，不急于产出执行任务。
- 项目方向讨论中，用户提出的长期原则、阶段边界、验收偏好和工程判断标准，可以沉淀进本文件，作为后续 SDD 文档编写的注意事项。

## 1. 基本原则

- 先理解项目上下文，再执行代码修改。
- 优先遵守项目已有架构、文档、接口与数据流。
- 不为了局部实现而破坏 Phase 1、Phase 2 已完成能力。
- 对不确定的项目事实，应先查阅仓库文件、脚本和文档。
- 不擅自删除、回滚或覆盖用户已有改动。
- 方案设计必须优先保证现有数据闭环可用：上传、raw 落盘、数据库入库、teacher API、dashboard 展示与验证脚本。
- 文档输出应具备可执行性，能被 CLI Codex 按步骤实施，而不是停留在口号或泛泛描述。

## 2. 开发边界

- 不修改 raw JSON 的既有结构。
- 不重构 Phase 1 上传链路。
- 不重构 Phase 2 teacher API。
- 不替换 PostgreSQL 主数据源。
- 不引入新的权限系统，除非用户明确要求。
- 不整体重写 dashboard，应在现有实现上小步增强。

## 3. 工作流程

- 开发前先确认相关 spec、tasks、status 文档。
- 修改代码后执行可行的静态检查或脚本验证。
- 涉及接口行为时，应给出 curl 或脚本验证结果。
- 涉及 dashboard 时，应说明浏览器验证方式或无法验证的原因。
- 完成任务后更新必要文档，尤其是 project-status、tasks 或变更记录。

## 4. 数据与兼容性

- raw JSON 是不可破坏的数据底座。
- 写入流程必须保持 raw JSON 落盘优先。
- 查询逻辑必须兼容字段差异，例如：
  - `score` / `overall_score`
  - `created_at` / `timestamp`
  - `event_type` / `type` / `category`
- 无筛选时允许 fallback 到文件或 sample。
- 有 `classroom_id`、`status` 等筛选时，禁止错误 fallback 到无关文件结果。

## 5. 输出要求

完成开发任务后，回复应包含：

- 修改文件列表
- 实现内容摘要
- 验证结果
- 未完成事项或遗留风险

## 6. 用户补充规则

用户后续提供的规则写入本节。

### 6.1 项目背景与产品定位

- 本项目目标是参加计算机设计大赛，并以获奖为导向进行规划。
- 项目目标用户主要是教师，同时需要管理员用户。
- 用户主要通过网站查看课堂数据、课堂分析结果和可视化展示。
- 项目整体链路为：树莓派端采集视频和关键帧，将 JSON 数据发送到本地端；本地端进行 YOLO 识别、行为分析和 session 整理；本地端再将结果发送到云服务器；云端负责数据存储、可视化展示和教师/管理员工作台。
- 当前开发重点是云端部分。
- 因为项目面向比赛展示，数据可视化不能只满足基础可用，还应尽可能体现视觉表现力、设计感、系统完整度和创新性。
- 后续进行流程设计、数据结构设计、接口设计时，应先确认不破坏既有数据流和数据结构，再结合比赛展示目标提升表达效果。

### 6.2 产品模块落地方案

- 产品统一定位为“智能课堂行为分析与教学反馈平台”。
- 系统采用统一登录入口，教师与管理员通过同一个登录页面进入系统，登录后按角色分流到不同端。
- 登录页应体现系统完整性和三端闭环，但不应做成空洞宣传页。
- 教师端的核心职责是“讲清楚一节课”，围绕单节课堂 session 的分析、录像、图表和反馈展开。
- 管理员端的核心职责是“证明这是一个平台”，围绕班级、教师、课堂数据、设备接入和系统状态展开。
- 比赛展示时应优先突出教师端课堂分析详情页，因为该页面最能体现采集、识别、分析、入库、可视化、反馈的完整闭环。
- 管理员端可以先做平台总览和基础数据管理，后续再逐步完善复杂管理能力。

推荐产品结构：

```text
统一登录
├── 教师端
│   ├── 今日课堂总览
│   ├── 课堂记录
│   ├── 课堂分析详情
│   │   ├── 课堂录像
│   │   ├── 注意力/活跃度时间线
│   │   ├── 教学阶段分布
│   │   ├── 区域表现
│   │   ├── 事件分布
│   │   └── 教学反馈摘要
│   └── 班级趋势分析
│
└── 管理员端
    ├── 平台总览
    ├── 班级管理
    ├── 教师管理
    ├── 全部课堂数据
    └── 设备与数据接入状态
```

- 阶段推进时，不应一次性实现全部模块；应优先完成最能支撑比赛展示的教师端课堂分析详情页，再补管理员端平台化能力。
