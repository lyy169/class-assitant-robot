# Cloud Git Convergence Plan

## 目标

在不影响当前运行中的部署目录前提下，恢复云端后续可 Git 管理的基础条件，并把“源码目录”和“部署目录”分离。

## 当前判断

`x:\video_project` 当前更像“部署副本 + 历史开发堆叠目录”，原因包括：
- 当前目录不存在 `.git`
- 同时混合存在旧 Flask 主应用、较新的 `cloud_backend/`、前端工程、运行数据库、上传目录、日志目录
- 目录中包含明显运行时资产：`instance/video.db`、`uploads/`、`logs/`、`cloud_backend/data/`

因此，不建议直接在当前目录初始化 Git 并将整个目录原样纳管。

## 推荐方案

### 方案结论

推荐采用“双目录收敛”：
- 当前 `video_project/` 继续作为部署副本保留
- 新建正式源码工作目录，例如 `video_project_src/`
- 在正式源码目录中重建 Git 管理
- 后续通过同步/部署脚本将正式源码目录中的主线资产发布到部署目录

### 为什么不直接在当前目录建 `.git`

直接在当前目录建 Git 的风险包括：
- 容易把运行数据、日志、数据库、上传文件直接纳入版本管理
- 容易在误操作时覆盖现网副本
- 难以清晰区分“源码变更”和“线上运行态变更”
- 后续做 legacy 隔离时会持续背负历史噪音

## 建议纳入正式源码目录的资产

第一批建议纳管：
- `cloud_backend/`
- `scripts/`
- `docs/`
- `frontend/`（如确认需要保留在线管理端）
- `backend/`（仅在确认 MP4 上传接口仍以此为主线时纳入）

第一批不应纳管的运行资产：
- `instance/`
- `uploads/`
- `logs/`
- `cloud_backend/data/`
- `__pycache__/`
- `frontend/node_modules/`
- `frontend/dist/`（除非明确需要构建产物随源码版本化）

## 收敛步骤建议

### 第 1 步：保持现部署目录不动

- 继续将 `video_project/` 视为运行目录
- 不在该目录直接恢复 Git
- 不移动数据库、上传目录、日志目录

### 第 2 步：建立正式源码目录

建议示例：

```text
/root/video_project          # 部署副本，继续运行
/root/video_project_src      # 正式源码目录，重新纳管 Git
```

正式源码目录初始化后，仅放源码、文档、示例配置、部署脚本。

### 第 3 步：迁入保留资产

优先迁入：
- `cloud_backend/`
- `docs/`
- `scripts/`

按确认结果再迁入：
- `frontend/`
- `backend/`

旧 Flask 主应用相关目录和运行资产不应在第一轮直接迁入主线源码目录。

### 第 4 步：建立同步机制

后续建议补一个明确的部署同步脚本，职责应仅限于：
- 从正式源码目录同步主线代码
- 保留部署目录中的 `.env`、日志、数据、数据库、上传目录
- 不执行强制覆盖运行数据的危险动作

推荐方向：
- `rsync` 白名单同步
- 或显式复制 `cloud_backend/`、`scripts/`、`docs/`
- 同步前输出变更清单

### 第 5 步：再做服务托管规范化

等正式源码目录稳定后，再推进：
- `cloud_backend` 的 systemd/supervisor 正式托管
- 旧 Flask 服务的 legacy 隔离
- 部署目录结构的进一步清理

## 上级或旁路 Git 线索

本轮检查结果：
- `video_project/` 当前不是 Git 仓库
- 其上级目录 `x:\` 未发现与本项目直接相关的 `.git` 线索
- 存在旁路目录 `x:\classroom-dashboard`，名称上可能与前端/仪表盘相关，但本轮未将其认定为当前部署目录的来源仓库

因此，目前没有证据表明可以安全地“恢复到某个现成上级 Git 仓库并覆盖当前目录”。

## 与现网共存原则

后续所有 Git 恢复动作应满足：
- 不 force push
- 不覆盖现运行目录
- 不重启现有服务
- 不在不确认来源的仓库中直接执行恢复覆盖
- 先文档化，再迁移，再部署
