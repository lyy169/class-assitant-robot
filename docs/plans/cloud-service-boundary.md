# Cloud Service Boundary

## 目的

本文档用于确认 `video_project` 当前云端正式保留的能力边界，避免在现运行目录中继续混合推进旧 Flask 主线与新的课堂互动分析云端接口。

当前结论遵循以下原则：
- 现部署目录继续保留并继续运行
- 新接口主线优先识别与文档化
- 旧 Flask 资产先保留，不直接删除
- 后续如做 Git 收敛，应先建立正式源码目录，再通过脚本同步到部署目录

## 云端正式主线

### 1. 课堂互动分析结果接收主线

正式主线入口为 `cloud_backend/`，当前职责是：
- 提供 `GET /health`
- 提供 `POST /api/interaction-results`
- 接收本地电脑推送的课堂互动统计结果 JSON
- 做最小必要字段校验
- 记录结构化日志
- 将原始结果按日期落盘到 `cloud_backend/data/raw/`

这部分是当前云端“新的正式接口部分”中最明确、最应保留的主线能力。

### 2. 树莓派 MP4 上传相关能力

`MP4` 上传仍属于正式保留需求，但当前不应删除或重构主链路。现阶段它主要体现在以下资产中：
- `uploads/`
- `backend/app.py` 中的 `/api/videos/upload`、`/api/videos/list`、`/api/videos/<id>/stream`
- `frontend/` 中的视频上传与列表页面

现阶段结论：
- 该链路属于正式需求
- 但它不是当前最清晰、最稳定的云端新主线实现
- 在第一轮整理中应标记为“保留资产，但暂不迁移、不切换、不删除”

## 职责区分

### 新接口主线

- `cloud_backend/`
  - 新的云端接收服务
  - 面向“本地 YOLO 统计结果推送到云端”
  - 是后续云端正式托管与 Git 纳管的首选核心目录

### 旧系统遗留

- `app.py`
  - 旧 Flask 主应用入口
  - 包含登录、视频页面、历史页面、录像状态等旧系统主线能力
- `templates/`
  - 旧 Flask 页面模板
  - 服务于登录页、视频展示页、测试页
- `static/`
  - 旧 Flask 静态资源目录
- `instance/video.db`
  - 旧 Flask/SQLite 本地数据库
  - 当前不迁移、不清理
- `README_DEPLOY.md`
  - 明显围绕 Flask + Gunicorn + Supervisor 的旧主线部署说明
- `gunicorn_config.py`
  - 旧 Flask 服务配置
- `models.py`
  - 旧 Flask 数据模型

以上内容退出云端正式主线，但具有归档价值，后续应进入 `legacy` 视角管理，而不是直接删除。

### 共用基础资产

- `uploads/`
  - 上传文件运行目录
  - 既与旧 Flask 视频链路相关，也与正式保留的 MP4 上传需求相关
  - 当前属于必须保留的运行资产
- `scripts/`
  - 服务器侧辅助脚本目录
  - 用于后续部署、验证、同步与托管辅助
- `docs/`
  - 项目规范、计划、变更记录
  - 属于正式保留资产
- `logs/`
  - 运行日志目录
  - 属于部署副本中的运行资产

## 保留 / Legacy / 待确认清单

### 保留

- `cloud_backend/`
- `scripts/`
- `docs/`
- `uploads/`

### Legacy

- `app.py`
- `templates/`
- `static/`
- `instance/video.db`
- `README_DEPLOY.md`
- `gunicorn_config.py`
- `models.py`

### 待确认

- `backend/`
  - 当前包含 Flask API，承担 MP4 上传与仪表盘接口
  - 其中 MP4 上传能力应保留
  - 但仪表盘统计接口仍偏演示性质，且与 `cloud_backend/` 主线职责不一致
  - 建议暂列为“待确认中的保留资产”
- `frontend/`
  - 当前主要对接视频列表与仪表盘接口
  - 与正式保留的 MP4 上传需求相关
  - 但是否属于云端服务器正式主线，还取决于后续是否继续作为在线管理端
- `config.py`
  - 服务于旧 Flask 主应用
  - 如仅用于旧主线，则后续可随旧系统并入 `legacy`
- `deploy.sh`
  - 当前看上去服务于旧 Flask 部署
  - 先保留，后续根据实际在线链路再决定是否归入 `legacy`

## 接口职责差异

### MP4 上传接口

MP4 上传接口面向文件传输，核心关注点是：
- 大文件上传
- 文件命名与保存
- 文件列表与在线播放
- 上传目录稳定性

这条链路偏“媒体文件管理”。

### 课堂互动结果 JSON 接口

课堂互动结果接口面向结构化分析结果，核心关注点是：
- JSON 校验
- API Key 或其他鉴权
- 日志与审计
- 原始结果归档
- 后续聚合分析与查询扩展

这条链路偏“分析结果接收与数据服务”。

两者都属于云端正式保留能力，但职责不同，不应在第一轮整理中被混为同一个重构目标。

## 第一轮处理边界

当前允许：
- 文档化主线与遗留边界
- 为 `cloud_backend/`、`scripts/` 补说明
- 设计后续 Git 收敛与部署规范

当前不允许：
- 删除旧 Flask 目录或文件
- 迁移数据库
- 迁移 `uploads/`
- 切断旧接口
- 修改现网服务托管方式
