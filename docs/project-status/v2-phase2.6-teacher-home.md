## 3. Added APIs

### `GET /api/teacher/overview`

Returns:

- `success`
- `teacher`
- `metrics`
- `latest_results`
- `classroom_summaries`
- `todo_items`

Metrics include:

- `classroom_count`
- `total_result_count`
- `recent_result_count`
- `raw_count`
- `reviewed_count`
- `archived_count`
- `avg_feedback_score`
- `avg_attention_score`
- `avg_response_score`

### `GET /api/teacher/results`

Query parameters:

- `classroom_id`
- `status`
- `days`
- `limit`
- `offset`

Returns:

- `success`
- `filters`
- `items`
- `total`

Each item includes:

- `result_id`
- `analysis_id`
- `classroom_id`
- `classroom_name`
- `lesson_title`
- `recorded_at`
- `generated_at`
- `created_at`
- `duration_seconds`
- `feedback_score`
- `attention_score`
- `response_score`
- `status`
- `has_video`
- `video_status`
- `detail_url`
- `updated_at`

Invalid `status` returns `400`. `days` accepts `7`, `30`, or `all`; numeric values are clamped to `7`, `30`, or all.

## 4. Data Aggregation

No core database tables were added.

Aggregation uses existing tables:

- `analysis_results`
- `sessions`
- `classrooms`
- future-compatible `users` context through `teacher_user_id`

When no auth context exists, the API uses a demo/default teacher context and aggregates visible results. If a bearer token is present and valid, `user_id` is passed through to repository aggregation.

Derived fields:

- `has_video` and `video_status` are inferred through Phase 2.5 video display mapping.
- `detail_url` points to `/dashboard?result_id={analysis_id}`.
- `todo_items` are generated by server-side rules and are not persisted.

Todo rules:

- raw results create a pending-review reminder.
- latest low attention result creates an attention reminder.
- high feedback result can create a highlight reminder.
- empty data creates an upload-waiting guidance item.

## 6. Dashboard Deep-Link

`GET /dashboard?result_id=xxx` is supported.

Behavior:

- If `result_id` exists, dashboard loads that result first through `GET /api/teacher/results/{result_id}`.
- If `result_id` is invalid, the dashboard displays a visible error and keeps the recent list available.
- If `result_id` is omitted, Phase 2.5 default behavior remains unchanged.

Implementation detail:

- `build_results_center_html()` now receives `selected_result_id`.
- The HTML exposes `data-initial-result-id`.
- The existing Vue app reads it on mount and fetches that result first.
